# システム改善点・運用上の課題レポート 2026-02-20

調査対象: `pages/master.php`, `pages/masters.php`, `pages/customers.php`, `pages/employees.php`, `pages/finance.php`, `pages/loans.php`, `pages/troubles.php`, `pages/photo-attendance.php`, `pages/payroll-journal.php`, `docs/architecture.md`, `config/config.php`

---

## 1. UI/UX 上の課題

### 1-1. 案件マスタ: ステータス列が一覧テーブルに存在しない

**現状**
`pages/master.php` のテーブル列は「案件番号 / バッジ / 現場名 / 営業担当 / ディーラー / 営業所 / 製品名 / サイズ」であり、ステータス列がない。ステータスを確認するには行を展開して詳細を開く必要がある。

**問題点**
最も頻繁に参照・変更されるはずのステータス（案件発生・成約・製品手配中・設置予定・設置済・完了）が一覧で見えない。フィルタードロップダウンで絞り込みはできるが、複数ステータスが混在している一覧を俯瞰できない。

**改善提案**
テーブルにステータス列（バッジ表示）を追加する。または行背景色をステータスに応じて変える（完了は薄グレー、未対応は薄赤など）。

---

### 1-2. 案件マスタ: 検索が P 番号・現場名のみでテキスト全文検索がない

**現状**
検索フォームは「P 番号で検索」「現場名で検索」の 2 項目のみ。顧客名・担当者名・製品名での検索はできない。

**問題点**
「〇〇社の案件を全部見たい」「〇〇製品の導入案件一覧を出したい」といった実務上よくある検索が不可能で、担当者がスクロールや目視で探す必要が生じる。

**改善提案**
検索ボックスを一本化し、P 番号・現場名・顧客名・担当者名・製品名を横断して部分一致検索するフリーワード検索に変更する。または詳細検索を折りたたみパネルに収めて現行の検索項目を補完する。

---

### 1-3. トラブル対応: 統計の担当者名がハードコーディング

**現状**
`pages/troubles.php` の統計サマリー部分（L406-415）で「足本」「曽我部」という担当者名が直接コードに記述されており、その 2 名の対応割合が棒グラフで表示される。

```php
$ashimotoCount = count(array_filter($data['troubles'] ?? array(), function($t) {
    return ($t['responder'] ?? '') === '足本';
}));
$sogabeCount = count(array_filter($data['troubles'] ?? array(), function($t) {
    return ($t['responder'] ?? '') === '曽我部';
}));
```

**問題点**
担当者が変わった場合・増えた場合にコード修正が必要になる。また特定の 2 名しか集計されないため、3 名以上が担当する場合に他の担当者の負荷が可視化されない。

**改善提案**
`troubleResponders` マスタから動的に担当者リストを取得し、全担当者の件数を集計・表示する。棒グラフも担当者数に応じて動的に生成する。

---

### 1-4. 顧客管理: 論理削除済み顧客が重複チェックに混入する

**現状**
`pages/customers.php` の重複チェック（L259-265）は論理削除済みの顧客を除外していない。

```php
foreach ($customers as $c) {
    if ($c['companyName'] === $companyName) {
        $exists = true;
        break;
    }
}
```

`$customers` は `getData()` 後に `decryptCustomerData()` した全データであり、`deleted_at` が付いているレコードも含まれる。

**問題点**
過去に論理削除した顧客と同名の会社を再登録しようとすると「既に登録されています」エラーになる。実運用では取引が再開する会社があるため、この制約が妨げになる。

**改善提案**
重複チェック時に `empty($c['deleted_at'])` を条件に加える。または論理削除済み顧客が存在する場合は「削除済みの同名顧客があります。復元しますか？」というダイアログを出す。

---

### 1-5. 従業員管理: 退職日が過ぎた従業員が有効扱いのまま残る

**現状**
`pages/employees.php` では `leave_date` を入力できるが、退職日を過ぎた場合に自動で非表示・無効化する処理がない。

**問題点**
退職済みの従業員が担当者選択リストや各種集計に含まれ続ける。photo-attendance の対象者リストなどでも退職者が表示される可能性がある。

**改善提案**
`filterDeleted()` に相当する `filterActiveEmployees()` 関数を整備し、`leave_date` が現在日より過去の場合はデフォルトで除外するようにする。一覧ページでは「退職者を含む」チェックボックスで切り替えられるようにする。

---

### 1-6. 財務管理: グラフが静的な棒グラフ実装でライブラリ不使用

**現状**
`pages/finance.php` のグラフ（`bar-chart` クラス）は CSS で高さを計算した `<div>` の棒グラフで、Chart.js 等のライブラリを使っていない。

**問題点**
ツールチップ表示・ズーム・月次推移の折れ線グラフなど、より詳細な分析に対応できない。現状のグラフは視認性は確保できているが、インタラクティブ性が低い。

**改善提案**
Chart.js（CDN 読み込み）を導入してグラフを置き換える。ホバー時の金額表示・複数月比較・取引先別内訳表示などを実現できる。

---

## 2. データ整合性の懸念

### 2-1. 案件マスタと顧客マスタの参照整合性がない

**現状**
`projects[].customer_name` は顧客のテキスト名を直接保持しており、`customers[].id` への参照ではない。

**問題点**
顧客名を `customers.php` で変更した場合、既存の `projects` に記録されている `customer_name` は更新されない。顧客名の表記揺れ（「株式会社〇〇」と「〇〇株式会社」）が発生した場合も同様に不一致が生じる。

**改善提案**
短期的には顧客名変更時に `projects` の `customer_name` を一括更新するスクリプトを整備する。中長期的には `projects[].customer_id` として ID 参照に移行し、表示時のみ名前を引くアーキテクチャに変更する。

---

### 2-2. トラブルと案件の紐付けがテキスト一致のみ

**現状**
`pages/troubles.php` では `pj_number` フィールドで案件マスタと紐付けるが、ID 参照ではなく文字列の大文字小文字を無視した一致（`strcasecmp`）に依存している（L646）。さらに P 番号で見つからない場合、案件名の部分一致検索にフォールバックする（L652-659）。

```php
// 見つからない場合、案件名で部分一致検索
if (!$projectInfo && mb_strlen($pjNumber) > 5) {
    foreach ($data['projects'] ?? array() as $proj) {
        if (mb_strpos($proj['name'] ?? '', $pjNumber) !== false || ...) {
```

**問題点**
案件名の部分一致検索は誤マッチのリスクがある。短い現場名（「千葉」など）が P 番号として入力されていると、全く別の案件にマッチしてしまう。また P 番号が存在しないトラブルが赤字で「未登録」表示されるが、どうすれば解決できるかユーザーに案内がない。

**改善提案**
トラブル登録フォームでは P 番号を datalist や select で選択式にし、フリーテキスト入力を廃止する。部分一致フォールバックは削除し、見つからない場合は「未登録」表示にとどめる。

---

### 2-3. product_category フィールドにIDと名前が混在

**現状**
マイグレーションを経ても `projects[].product_category` にはカテゴリ ID が入るケースと、カテゴリ名が直接入るケースが混在している。`$categoryMap` に名前→名前のマッピングを追加することで表示は対応されているが、データ側の正規化は未完了。

**問題点**
今後 `product_category` をキーに集計・フィルタリングを行う際に、ID と名前の混在が原因で誤った集計が生じる可能性がある。CSVエクスポートや外部連携でも同様の問題が起きうる。

**改善提案**
全 `projects` の `product_category` をカテゴリ ID に統一する完全マイグレーションを実施する。マイグレーション後は `$categoryMap` の名前→名前マッピングを不要化し、ID のみで運用する。

---

### 2-4. 給与仕訳: 社員番号が固定値でハードコーディング

**現状**
`pages/payroll-journal.php` の L5 で役員判定用の社員番号が直接コーディングされている。

```php
$EXECUTIVE_EMPLOYEE_NO = '000001';
```

**問題点**
役員が変更になった場合・複数役員が存在する場合にコード修正が必要になる。仕訳の誤分類（役員報酬 vs 給料賃金）は税務上の問題になりうる。

**改善提案**
従業員マスタに `is_executive` フラグを追加し、役員判定をデータドリブンにする。または `config.json` に役員社員番号の配列として外出しする。

## 3. 権限・セキュリティの懸念

### 3-1. 顧客データの暗号化と平文混在

**現状**
`functions/encryption.php` による顧客データの暗号化が実装されており、`decryptCustomerData()` / `encryptCustomerData()` で読み書きされている。ただし `masters.php` の顧客処理では `encryptCustomerData()` が呼ばれているが（L58）、`saveData()` のインデント位置がズレており（L58の直前で暗号化、L58でsaveData）、コードの見通しが悪い。

**問題点**
暗号化忘れによる平文保存のリスクがある。`customers.php` と `masters.php` の両方に顧客処理が存在し（互換性のため残留）、どちらが正とも言えない重複実装になっている。

**改善提案**
`saveData()` 内で顧客データを自動的に暗号化する（暗号化を saveData のオプションとして組み込む）か、`saveData()` 呼び出し前に `encryptCustomerData()` を呼ぶことをドキュメントで明確に義務付ける。`masters.php` の顧客処理は削除して `customers.php` に一本化する。

---

### 3-2. sales ロールの編集禁止範囲が不明瞭

**現状**
権限システムは `sales < product < admin` の 3 段階（`config/config.php` L64）。`sales` ロールは閲覧のみとされているが、実際には各ページで `canEdit()` によるガードが異なる。

- `loans.php`: `if (!canEdit()) header('Location: index.php'); exit;` でページ全体をブロック
- `master.php`: 削除ボタンは `canDelete()` でガード、編集ボタンは全員表示されている（POST 受信時のみ権限チェック）

**問題点**
`sales` ロールのユーザーが案件マスタで編集ボタンをクリックするとモーダルが表示されてしまう（実際に POST しても 403 が返るが、UX として不整合）。

**改善提案**
編集ボタンの表示も `canEditCurrentPage()` で条件分岐し、`sales` ロールには編集ボタン自体を非表示にする。

---

### 3-3. セッションエラーメッセージの XSS リスク

**現状**
`pages/troubles.php` の L359 でセッション経由のエラーメッセージを `htmlspecialchars()` なしで出力している。

```php
<?php if (isset($_SESSION['error_message'])): ?>
    <div class="alert alert-danger">
        <?= $_SESSION['error_message'] ?>
    </div>
```

**問題点**
セッションは原則サーバーサイドで管理されるため通常は問題ないが、将来的にセッションへの書き込み元が増えた場合（外部APIレスポンスをそのままセッションに格納するなど）にXSSのベクタになりうる。ガイドラインとして一貫性が欠ける。

**改善提案**
全てのセッション経由メッセージ出力に `htmlspecialchars()` を適用する。

---

## 4. 機能の欠如

### 4-1. 案件マスタにステータス変更履歴がない

**現状**
案件の更新は `auditUpdate` / `writeAuditLog` で記録されるが、ステータス変更の前後値まで追跡するビューはない。

**問題点**
「この案件はいつ『成約』になったか」「誰が『完了』にしたか」を確認する手段がなく、プロジェクトの進行遅延や担当者の作業状況を振り返れない。

**改善提案**
案件詳細パネルにステータス変更履歴タブを追加し、`audit-log` から当該案件の変更ログを抽出して表示する。

---

### 4-2. トラブル対応に添付ファイル機能がない

**現状**
`troubles.php` のトラブル登録フォームにはテキストフィールドのみで、写真・PDF などの添付機能がない。

**問題点**
現場でのトラブル状況を写真で記録したい場合、別の経路（Google Chat 等）で共有することになり、トラブル管理システムとの連携が分断される。

**改善提案**
Google Drive との連携機能（すでに `loans.php` で実装済み）を応用し、トラブルに対して Drive ファイルのリンクを紐付けられるようにする。あるいはシステム内ファイルアップロードを実装する（ただしサーバー容量管理が必要）。

---

### 4-3. 財務管理に予算・予実管理機能がない

**現状**
`finance.php` は MoneyForward からの請求書データを同期・表示するが、月次予算の登録・予実比較機能がない。

**問題点**
売上実績はMF同期で確認できるが、「今月の目標売上に対して何%達成したか」がシステム上で把握できない。

**改善提案**
月次予算テーブルを `data.json` に追加し、finance ページで予算入力フォームと予実グラフ（目標 vs 実績）を追加する。

---

### 4-4. 貸出管理にリマインダー・期限管理機能がない

**現状**
`loans.php` は借入先・返済データの管理と Google Drive 連携による PDF 突合機能を持つが、次回返済日の通知・リマインダー機能がない。

**問題点**
返済日を手動で把握・管理する必要がある。返済漏れのリスクがある。

**改善提案**
次回返済日が近づいた場合に Google Chat へ通知を送る仕組みを追加する（既存の `GoogleChatClient` を活用）。

---

### 4-5. 写真勤怠管理: 土日祝日の対応が未整備

**現状**
`pages/photo-attendance.php` の稼働日数カウントは月〜金（weekday 1-5）のみで計算されており（L86-89）、祝日は考慮されていない。

```php
for ($d = $monthStart; $d <= $todayOrEnd; $d = date('Y-m-d', strtotime($d . ' +1 day'))) {
    $dow = date('N', strtotime($d));
    if ($dow <= 5) $workingDaysSoFar++; // Mon-Fri
}
```

**問題点**
祝日が含まれる月は稼働日数が過大カウントされ、提出率が実態より低く計算される。提出率 100% を達成しているのに数値上では不足に見えてしまう。

**改善提案**
祝日マスタ（JSON ファイルまたは外部 API）を導入し、稼働日数計算から祝日を除外する。短期的には `config/holidays.json` に年間祝日リストをメンテナンスする方法が現実的。

## 5. 運用上の手間

### 5-1. MF 請求書同期が手動トリガー

**現状**
`pages/finance.php` では `$autoSyncEnabled = false;`（L48）と明示的に自動同期が無効化されており、同期は手動ボタン押下でのみ実行される。

**問題点**
毎日または週次で担当者が手動で同期ボタンを押す必要があり、押し忘れると財務データが古くなる。

**改善提案**
PHP の `cron` に相当する処理（Windows の場合はタスクスケジューラ）で `api/mf-sync-task.php` を定期実行するスクリプトを整備する。または `autoSyncEnabled` を `true` に変更し、最終同期から X 時間経過していれば自動同期する既存のコードを有効化する。

---

### 5-2. スプレッドシート同期後の差分確認手段がない

**現状**
`master.php` のスプシ連携では同期後に「追加: N件, 更新: N件」のカウントのみ表示される。どのレコードが変更されたかは確認できない。

**問題点**
スプレッドシートの誤入力がシステムに反映された場合、件数だけでは検知できない。変更内容を確認するには audit log を参照するしかなく、操作が煩雑。

**改善提案**
同期後に変更されたレコードの差分プレビュー（更新前・更新後）をモーダルで表示し、確認・承認後に保存する方式に変更する。

---

### 5-3. 給与仕訳の仕訳マッピングがコードに埋め込まれている

**現状**
`pages/payroll-journal.php` の `$JOURNAL_MAPPINGS` 配列（L8-93）で勘定科目・補助科目・税区分が定義されており、変更にはコード修正が必要。

**問題点**
会計ソフトの設定変更・勘定科目名の変更があった場合、開発者でないと対応できない。

**改善提案**
`config/journal-mappings.json` として外出しし、管理画面から編集できるようにする。

---

### 5-4. 担当者（assignees）が案件マスタと各種マスタで二重管理

**現状**
担当者は `data.json` の `assignees` で管理されているが、`master.php` からも `masters.php` からも追加・削除できる。`master.php` 側の担当者追加（L422-426）はシンプルな ID 採番で、`masters.php` 側（L133-149）では `uniqid()` 形式と異なる。

**問題点**
担当者 ID の形式が inconsistent（整数 vs `a_xxxx`）で、今後フィルタや集計を ID ベースで行う際に問題が生じる可能性がある。

**改善提案**
担当者管理を `masters.php` に一本化し、`master.php` からの直接追加は廃止する。ID 形式も `a_` プレフィックス形式に統一する。

---

## 6. パフォーマンス

### 6-1. 全データをメモリ上に展開してから絞り込む設計

**現状**
`getData()` は `data.json` 全体をメモリに読み込み、PHP の配列として保持する。案件マスタ・トラブルなどの絞り込みはすべてインメモリのフィルタリング。

**問題点**
現時点では案件数 299件・トラブル数も数百件程度で問題ないが、将来的に数千件規模になった場合、毎リクエストごとに MB 単位の JSON を読み込む形は非効率になる。特に `troubles.php` では全 `data.json` を読み込んだ後、配列フィルタを複数回掛ける。

**改善提案**
短期的には `getData()` の static キャッシュ（既実装）で同一リクエスト内の重複読み込みは防げている。中長期的には SQLite などの軽量 RDBMS への移行を検討する。移行コストが高い場合は、エンティティ別に JSON ファイルを分割する（`data-projects.json`, `data-troubles.json` など）。

---

### 6-2. トラブル一覧で全案件データを O(n×m) 検索

**現状**
`pages/troubles.php` のトラブル一覧では、各トラブル行の P 番号を案件マスタから検索するために O(n) のループを毎回実行している（L644-659）。トラブル件数 × 案件数 の組み合わせになる。

**問題点**
現時点は件数が少なく体感速度に影響はないが、件数が増えると線形に重くなる。

**改善提案**
案件マスタを P 番号をキーにした連想配列（`$projectsById`）として事前に構築しておき、O(1) で参照する。PHP での実装は容易で効果が大きい。

---

## 7. エラーハンドリング

### 7-1. Google Drive / Sheets 接続エラーが汎用メッセージのみ

**現状**
`pages/loans.php` では `try-catch` で Google Drive 接続エラーを捕捉しているが（L200-202）、エラーメッセージは `'Google Drive接続エラー: ' . $e->getMessage()` のみ。

**問題点**
ユーザーには詳細なエラー理由（OAuth トークン切れ・権限不足・ネットワーク障害）が判断できず、再認証が必要なのかシステム管理者に連絡すべきなのかが不明。

**改善提案**
エラーの種類に応じたメッセージとアクション（「再認証が必要です → ここをクリック」等）を分岐して表示する。OAuth 期限切れは最も多いエラーなので優先的に対応する。

### 7-2. MF API 同期失敗時のロールバックがない

**現状**
`pages/finance.php` の MF 同期処理（L62-135）では、`syncMfInvoices()` が失敗した場合でも `saveData()` の前にエラーをキャッチして `header('Location: ...')` でリダイレクトするが、中途半端に `$data` が更新された状態で `saveData()` が呼ばれるリスクがある。

**問題点**
MF API から途中まで取得・同期した状態でエラーが発生した場合、不完全なデータが保存される可能性がある。

**改善提案**
同期処理の前後でデータのスナップショットを明示的に取り、失敗時はスナップショットから復元するトランザクション的な処理を追加する。`createAutoSnapshot()` は既に実装されているので、復元処理（`restoreFromSnapshot()`）を追加する。

### 7-3. POST 失敗時のフォーム入力値が失われる

**現状**
多くのページでは POST 処理後に `header('Location: ...')` でリダイレクトするため、バリデーションエラーの場合もフォームの入力値がリセットされる。例: `customers.php` でメールアドレスのバリデーションエラー時（L254）。

**問題点**
長いフォームで入力途中にバリデーションエラーが出ると、全項目を入力し直す必要があり、特に案件登録モーダルのような多項目フォームでは顕著な手間になる。

**改善提案**
バリデーションエラー時はセッションに入力値を保存してリダイレクト後に復元する（PRG パターンの拡張）。または AJAX で送信してエラーを JSON で受け取り、JS でエラー表示する方式に移行する。
